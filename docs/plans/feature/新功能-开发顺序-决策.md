## 1. 问题陈述

用户希望评估三个新功能的开发顺序:

1. **图片编辑** — 对已有图片做风格化、改动、增强
2. **LoRA 训练** — 让用户用自己的素材训练个性化 LoRA
3. **节点编辑** — 画布式编辑器,用于长视频制作,可生成视频素材和文本

核心问题: 三者应该按什么顺序开发,如何避免精力分散和路径冲突?

## 2. 现状审计 (基于代码扫描)

## 3. 核心前提 (Premises)

### P1: 视频剧本(线性) 与 节点编辑(非线性) 目标重叠

- 项目已经有 5 阶段视频剧本规划: 主题 → 剧本 → 分镜 → Clip → 拼接
- 节点编辑器是 ComfyUI / TouchDesigner 风格的可视化编程范式
- **两者目标相同(做长视频),路径相反**。并行做会精力分散
- 视频剧本对普通创作者友好;节点编辑只对重度技术创作者友好

### P2: LoRA 训练强依赖素材管理

- 没有素材集就训不了 LoRA
- F20 Asset Library 被项目自己 Parked,说明对完整版 ROI 存疑
- 必须二选一: 同时做完整素材库 (工作量翻倍) 或 用"现场上传"轻量方案

### P3: LoRA 训练 → 图片编辑 存在杠杆复利

- `image-transform.service` 的 Phase 3 `detail` transformation 设计签名是 `params: { loraId, strength }`
- LoRA 训完后用户可以在图片编辑里"用我自己的风格变体",并在 generate-image 链路直接使用
- 两者是配套关系,做完会形成 1+1>2 的产品体验

### P4: 节点编辑是高风险投入,应该被用户数据决策驱动

- 画布编辑器开发成本高 (节点引擎 + 拖拽 + 端口连接 + 实时预览 + 视频拼接)
- 在视频剧本上线、收集真实用户反馈前,无法判断"用户真的需要自由组合素材"
- 直接做节点编辑器是基于直觉的下注,不是基于证据的决策

## 3.5 主流 AI 长视频生成路线 (2026 市场调研)

经 web 调研,主流 AI 长视频生成工具分四派:

| 流派            | 代表产品                                | 产品形态                                                                  | 目标用户      |
| --------------- | --------------------------------------- | ------------------------------------------------------------------------- | ------------- |
| 线性 Storyboard | Sora Storyboard / Runway Stories / Pika | 时间线 + 每镜头独立生成 → NLE 拼接                                        | 编剧 / 导演型 |
| 节点画布        | **Martini** / **Bonzi** / Vilva / Krea  | 画布 + 多类型节点(剧本/分镜/角色/音频/视频)+ 工作流复用                   | 创意制作团队  |
| AI Studio 画布  | **即梦 Seedance 2.0** / 可灵 Kling 3.0  | 画布 + Agent 对话 + 多模态参考(9 图 + 3 视频 + 3 音频)+ 主体库 + 自动分镜 | 短剧生产团队  |
| 技术派节点      | ComfyUI / InvokeAI                      | 完全自由的节点图                                                          | 技术创作者    |

**关键证据**:

- 即梦 2026 日营收 7000 万 RMB,AI 短剧已超真人短剧;形态是"画布 + Agent + 多镜头 + 主体一致性"
- Martini: "production teams storyboard mode, script nodes, lip-sync, plan scenes, generate keyframes, fan out video models, assemble narrative sequence from one canvas" — 与用户描述高度对齐
- Bonzi 的 "Project Brain" 跨镜头记忆 = 角色 / 风格 / 品牌一致性

**用户"节点编辑"的真实定位**: 不是 ComfyUI 风格,而是即梦 AI Studio / Martini / Bonzi 一类的"预定义节点 + 画布 + 拼接"工作台。

| 维度                        | 即梦炉米 Lumi (企业版) | 可灵主体参考 (C 端) | PixelVault LoRA (规划) |
| --------------------------- | ---------------------- | ------------------- | ---------------------- |
| C 端可用                    | ❌                     | ✅                  | ✅                     |
| 训开源模型 (FLUX/SDXL)      | ❌                     | ❌                  | ✅                     |
| LoRA 可导出 / 跨工具用      | ❌                     | ❌                  | ✅                     |
| 可训风格 / 物体 / 场景      | ✅ (企业)              | ❌                  | ✅                     |
| BYOK 模式                   | ❌                     | ❌                  | ✅                     |
| 跨工具复用 (画布/编辑/生成) | ❌                     | ❌                  | ✅                     |

**核心差异化护城河**: C 端 + BYOK + 开放 LoRA + 跨工具复用。

**风险**: 即梦/可灵的"参考图路线"对普通用户已够用,LoRA 训练的差异化要讲给对的人听:

- 创作者
- 动漫党
- IP 持有者
- 想做长期角色 IP 的用户

**GTM 策略**: LoRA 训练完成后,在 UI 上展示"你的 LoRA 已可用于: 图片生成 / 图片编辑 / 视频生成 / 节点画布 / 一键导出",把跨工具复用做成护城河可视化。

## 3.6 节点编辑的依赖拆解

节点编辑画布的核心不是节点编辑器本身,而是**底层素材生成能力 + 一致性 + 拼接**:

| 节点类型           | 依赖的底层能力                        | 当前完成度 |
| ------------------ | ------------------------------------- | ---------- |
| 剧本节点           | 视频剧本服务 (已规划但未实施)         | 0%         |
| 三视图节点         | 图片编辑(姿态变换) + LoRA(角色一致性) | 40% + 25%  |
| 背景节点           | 文生图(已有) + 图片编辑(风格化)       | 70% + 40%  |
| 角色节点           | Character Card(已有) + LoRA           | 80% + 25%  |
| 声音节点           | TTS + voice cloning (已有)            | 80%        |
| 视频片段节点       | generate-video (已有)                 | 80%        |
| 拼接节点           | 视频剧本 Phase 4                      | 0%         |
| 画布 UI + 编排引擎 | 全新                                  | 0%         |

**结论**: 节点编辑是顶层 UI 整合;底层依赖恰好是另外两个功能 + 视频剧本规划。**做节点编辑 = 做完底层 + 做画布 UI**,不存在"先做节点编辑跳过其他"的选项(除非接受输出质量被即梦 / Martini 碾压)。

## 4. 已考虑的 Approach

### Approach A: 收尾驱动 (推荐)

**顺序**: 图片编辑收尾 → LoRA 训练(含轻量素材) → 视频剧本 → (再评估)节点编辑

- 图片编辑(2-3 周): 完成 Strategy Pattern 剩余 handler,UI 接入现有 IMAGE_EDIT_REMIX workflow
- LoRA 训练(5-7 周): 用"现场上传 10-30 张图 + 简易 tagging"替代完整 Asset Library,接入 FAL/Replicate flux-lora-trainer,训练完自动注入图片编辑 detail handler + generate-image lora capability
- 视频剧本(6-8 周): 按已有 Phase 0-4 规划推进
- 节点编辑: 暂缓,等视频剧本上线后用真实反馈决定

总时长: 13-18 周

**优点**:

- 每一步都是低风险、高复用、明确收益
- 图片编辑 + LoRA 形成杠杆复利
- 视频剧本规划成熟,工作量可控
- 避免节点编辑"无底洞"

**缺点**:

- 节点编辑被无限期延后
- 总时长 13-18 周,需要耐心

### Approach B: 长视频需求优先

**顺序**: 视频剧本 → 图片编辑 → LoRA 训练 → (节点编辑可选)

如果判断"长视频"是当前最痛用户需求,先打这个市场。

**优点**:

- 长视频是当前 AI 创作工具高需求战场
- 视频剧本规划已 ready

**缺点**:

- 6-8 周内无新功能上线,用户感知慢
- 浪费了图片编辑 40% 已完成工作
- LoRA + 图片编辑杠杆复利被延后

### Approach C: 创作者重度路线 (ComfyUI 风格)

**顺序**: LoRA 训练 + 完整 Asset Library → 节点编辑 → 图片编辑(顺便)

押注重度创作者市场,做成 ComfyUI 级别专业工具。

**优点**:

- 上限最高

**缺点**:

- 工作量超过 20 周,中间无可上线节点
- Asset Library 是项目自己 Parked 的功能,ROI 存疑
- 节点编辑开发成本极高
- 用户面变窄 (但你说的"节点编辑"其实不是这一类,所以这个 Approach 实际不匹配)

### Approach A' (修正版): 底层先行 + 节点编辑作为终极拼图 ⭐

**顺序**: 图片编辑收尾 → LoRA + 轻量素材 → 视频剧本 Phase 1-3 → 节点编辑画布 + 拼接

逻辑变化: 节点编辑**不是被延后**,而是作为底层能力齐备后的**顶层 UI 整合**。

- Stage 1: 图片编辑收尾 (2-3 周) — 解锁三视图节点 / 背景节点能力
- Stage 2: LoRA + 轻量素材 (5-7 周) — 解锁主体一致性
- Stage 3: 视频剧本 Phase 1-3 (5-6 周) — 解锁剧本节点 / 分镜节点 / clip 引擎,不做 Phase 4 拼接(留给节点编辑统一做)
- Stage 4: 节点编辑画布 + 视频拼接 (8-12 周) — 终极形态

总时长 20-28 周,但每一步都可上线,节点编辑最终形态非常完整。

**优点**:

- 节点编辑有明确目标
- 每一步都为终极目标铺路,不浪费
- 底层 ready 后节点编辑能直接出 Martini / 即梦 级别的产品

**缺点**:

- 节点编辑形态在 20+ 周后才能让用户看到
- 期间 UX 决策都是基于猜想

### Approach E (新方案): 双线并行 — 底层 + 节点 prototype ⭐⭐

**主线**: 同 A' (底层能力按顺序推进)

**副线 (第 5-7 周启动)**:

- 节点编辑 prototype: 最简版画布 + 4-5 个节点类型 + 只能输出 30s 短视频
- 用现有 generate-image / generate-video / TTS 作为节点能力
- 仅用于验证画布 UX 和节点交互范式
- 不追求长视频质量,只追求"节点编排能跑通"

**整合期 (第 15-20 周)**: 把图片编辑 / LoRA / 视频剧本能力填充到节点

**终态 (第 20-28 周)**: 完整长视频画布,对标 Martini / 即梦 AI Studio

**优点**:

- 第 5 周左右就有节点编辑形态可见,持续迭代 UX
- UI 团队和底层团队可以并行
- 节点编辑不会"等到最后才上线",降低你对节点编辑被延后的焦虑
- 早期 prototype 收集真实用户反馈,指导后续节点设计

**缺点**:

- 需要至少 1.5 个开发人力 (主线一人,副线半人)
- 如果只有 1 人,会精力分散
- prototype 阶段输出质量低,需要明确标记 "Beta / Lab"

### Approach F: 节点编辑 First — 用现有能力

跳过图片编辑 / LoRA,直接做节点画布,用 generate-image / generate-video / TTS 当节点。

**优点**: 节点编辑最快上线 (8-12 周)

**缺点**:

- 角色一致性差(没有 LoRA + 三视图编辑)
- 视频质量上限低,会被即梦 / 可灵 / Martini 碾压
- 图片编辑那 40% 已完成的工作浪费
- 节点编辑做出来后用户会要求"我想固定角色样子",这时反推回去做 LoRA,工程上更乱

### Approach G (用户澄清后的修订版): 素材工作台 → 节点画布 → LoRA 后置 ⭐⭐⭐

**用户策略澄清**: 角色生成主力是 GPT-Image / Gemini Pro Image, 一致性靠"详细介绍 + 三视图 + 服装",LoRA 后置作为辅助。

**顺序**:

| Stage | 周次   | 内容                                                                             | 解锁                                                       |
| ----- | ------ | -------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| 1     | W1-4   | 图片编辑升级为"角色素材工作台" (三视图自动生成 + 服装变换 + 多视角背景 + 风格化) | 节点画布的 CharacterNode / BackgroundNode / StyleNode 底层 |
| 2     | W5-10  | 视频剧本 Phase 1-3 (剧本 / 分镜 / clip)                                          | 节点画布的 ScriptNode / SceneNode / ClipNode 底层          |
| 3     | W11-22 | **节点编辑画布 + 视频拼接** (终极目标)                                           | 完整产品                                                   |
| 4     | 后置   | LoRA 接入 (Civitai 公开 LoRA hub) 或 LoRA 训练 (按市场反馈决定)                  | 差异化护城河强化                                           |

总时长 20-22 周,比 Approach E 少 6 周。

**优点**:

- 节点编辑画布的优先级提升,与用户最想要的对齐
- 三视图工作台是真差异化,即梦/可灵/Martini 都没有做精
- LoRA 后置释放 5-7 周时间投入节点画布精细化
- 与 BYOK 模式一致 (用户可选用哪个强模型)

**缺点**:

- 质量上限受 OpenAI / Google 模型限制(黑盒)
- BYOK 成本提示 UI 必须做好,否则用户体感"烧钱"
- 没 LoRA 训练,长期护城河弱,需要靠工作流深度和品牌建立壁垒
- 6 个月后即梦若 C 端开放 LoRA,需要快速跟进

**反向建议: Stage 4 做"LoRA 接入"而非"LoRA 训练"**

- 接入 Civitai 等公开 LoRA hub (省训练基础设施)
- 立即让画布支持万级公开 LoRA
- 节点画布的角色节点/风格节点都增强
- 用户真要训自己 LoRA, 可以本地训完上传

### Approach H (工作流驱动版,基于真实创作者工作流调研) ⭐⭐⭐⭐

**调研结论**: 主流 AI 视频创作者 6 步法 = 剧本 → 分镜 → 角色一致性(参考图绑定) → 视频生成(i2v) → 配音/对口型 → 后期剪辑。一站式平台是 2026 终极形态(有戏 AI / Vidu Q3 / Zopia)。**角色资产库 / 主体库** 是新标准,介于参考图和 LoRA 之间。

**核心策略**: 升级现有 Character Card (80% 已完成) 为"角色资产库",围绕它构建全工作流。

**顺序**:

| Stage | 周次   | 内容                                                                    | 对标                | 主要工作                                                                    |
| ----- | ------ | ----------------------------------------------------------------------- | ------------------- | --------------------------------------------------------------------------- |
| 1     | W1-4   | Character Card 升级为"**角色资产库**" + 图片编辑收尾                    | Vidu Q3 主体库 2.0  | 三视图 / 服装变体 / 表情包 / 姿态库 / 场景预览; image-transform 收尾        |
| 2     | W5-10  | 视频剧本服务 — 线性版本 (剧本 + 自动分镜 + i2v + 配音)                  | 有戏 AI / Jellyfish | 完整线性创作流程, 用户能从主题一键到 1-2 分钟视频                           |
| 3     | W11-22 | **节点画布 + 视频拼接** — 一站式工作台 (线性流程的高级模式, 同底层能力) | Martini / Zopia     | CharacterAssetNode / ScriptNode / StoryboardNode / ClipNode / StitchNode 等 |
| 4     | 后置   | LoRA 接入 Civitai → 多模态参考(对标 Seedance 2.0) → LoRA 训练           | 即梦炉米 / Seedance | 按市场反馈逐步加强差异化                                                    |

总时长 22-26 周。

**独特价值**:

1. **完全对齐主流参考图绑定法** — 不押注 LoRA, 技术风险低
2. **复用 Character Card (80% 已完成)** — 升级为竞品级角色资产库,不浪费已有投入
3. **线性流程先跑通 (Stage 2)** — 用户先能用线性向导出短视频, 验证 pipeline
4. **节点画布是线性流程的高级模式** — 普通用户用线性,高级用户用画布,**同一套底层能力**, 工作量大幅压缩
5. **定位清晰**: 节点画布对标有戏 AI / Vidu / Zopia, 不是 ComfyUI

**风险**:

- 参考图绑定法的质量上限受 i2v provider 决定 (Seedance 2.0 / 可灵 / Runway), 需要 multi-provider 保底
- 一站式平台赛道竞争激烈, 有戏 AI / Zopia / Vidu 都在跑, 6 个月内格局可能变化
- 角色资产库做精比做全更重要, 否则就是"另一个 Character Card"

## 5. 推荐 Approach: H (工作流驱动版)

**理由**:

1. **完全基于真实创作者工作流调研**, 不是 PixelVault 团队的预想
2. **角色资产库**是 2026 新标准(Vidu Q3 主体库 / 即梦全能参考), 你的 Character Card 已经做了 80%, 升级它就立刻拥有竞品级能力
3. **参考图绑定法 (i2v) 是主流路径**, 不需要 LoRA, 技术风险最低
4. **Stage 2 线性版本先跑通完整 pipeline**, Stage 3 节点画布只是 UI 升级, 共享同一套底层 — 工作量被腰斩
5. **节点画布定位清晰**: 不对标 ComfyUI, 对标有戏 AI / Vidu / Zopia / Martini — 一站式 AI 视频创作平台
6. **LoRA 真后置** — 主流创作者已经证明"参考图 + 角色资产库"够用, LoRA 是高级辅助, 等用户反馈再做

## 6. Approach H 详细执行计划 (基于创作者工作流)

### 6.0 主流 AI 视频创作者工作流 (调研结论)

经典 6 步法 (中小创作者主流):

```
1. 剧本生成 (LLM: 豆包/GPT/Claude)
    ↓
2. 分镜设计 (文字描述 → AI 分镜图: 即梦/Midjourney/Flux)
    ↓
3. 角色一致性 (参考图绑定法 ⭐ 主流) — @图片 绑定身份
    ↓
4. 视频生成 (即梦/可灵: 首帧 + 图生视频 + 5s/10s)
    ↓
```

### 6.2 NEW: 视频剧本服务 — 线性版本 (Week 5-10)

**目标**: 让用户从主题一键生成 1-2 分钟视频, 跑通完整 pipeline. 这是节点画布 Stage 3 的能力底层.

**工作包**:

A. 剧本生成服务 (LLM)

- Prisma `VideoScript` + `VideoScriptScene` 模型 (已规划)
- `video-script.service` — 输入主题/网文/IP, 输出结构化剧本
- 场景反推: 按总时长 → 场景数 → 每场景 duration
- API: POST /api/video-script + PATCH /api/video-script/[id]

B. 自动分镜

- 每个 scene 调 generate-image, **用角色资产库作为 reference 锁定身份**
- 多视角 / 多场景批量生成
- 失败重试 + 单格替换
- UI: SceneFrameGrid

C. i2v 视频片段生成

- 用分镜帧作为首帧, 走现有 generate-video
- Provider 选型: Seedance 2.0 Fast / 可灵 / Runway, 用户可切
- 每个 clip 5-10 秒

D. 配音生成 (复用现有 TTS)

- 用户选择 voice → 整个剧本一次 TTS

E. 拼接 (轻量 MVP 版)

- 不做转场, cut-to-cut
- 长视频 (>60s) 走服务端 FFmpeg

F. 线性向导 UI

- 5 步向导: 主题 → 角色 → 剧本 → 分镜确认 → 生成
- 每步都可回退修改
- 进度可见, 错误降级

**完工标准**: 用户输入"猫咪和狗狗的友谊" → 选角色资产 → 60 秒内拿到 8 个分镜 → 90 秒内拿到 8 个 clip → 自动拼接 + 配音 → 1 分钟视频成片. **80% 用户能 5 分钟内完整跑通**.

- **缓解**: 多 Provider 接入 (Gemini / GPT / Seedream / Imagen 等),Provider 出问题快速切换

**风险 2: BYOK 模式与高 token 消耗的张力**

- 一段 1 分钟视频 ≈ 35 次生成 ≈ $1.4+视频成本
- 用户的 API key 账单会比想象的高
- **缓解**: 每个节点显示预估成本; 整个画布显示累计预估; 生成前必须 confirm

**风险 3: 没 LoRA 的差异化护城河弱**

- 即梦/可灵/Martini 都能调强模型 API
- 你目前的差异化: 画布形态 + BYOK + 三视图工作流 + 跨工具复用
- 画布形态对手 1-2 个月就能抄
- **缓解**: 三视图工作流做到极致 (交互 / 模板 / 效果验证 / 角色 IP 持续性); 节点画布积累用户工作流模板,形成网络效应

**风险 4: LoRA 后置 = 接受工作流公司定位**

- 6 个月后即梦若对 C 端开放 LoRA,你的"LoRA 待办"就晚了
- **缓解**: Stage 4 用"LoRA 接入 Civitai" 快速建立 LoRA 生态(不做训练),保留 LoRA 训练能力的快速启动入口(代码层面 lora capability 已存在);市场信号触发立即启动训练

**风险 5: Stage 3 节点画布工作量被低估**

- 12 周做完整画布 + 拼接 + 6+ 节点类型 + 一致性引擎,工作量可能不止
- **缓解**: Week 11 做完详细工作分解,如果发现 >12 周,先做 MVP 6 节点 + 短视频拼接(60s),长视频 Phase 2 再上

### 关键监控指标

- Week 12: 重新审视即梦 / 可灵 / Martini 的市场动作,决定 Stage 3 工作包优先级
- Week 16: 节点画布 MVP 上线时, 监控用户对"三视图工作台"的留存,如果留存好,LoRA 后置正确;如果留存差,提前启动 LoRA 接入

## 11. What I noticed about how you think

- 你提的三个功能本质都是**增加创作自由度**(图片编辑改图、LoRA 训练自己的风格、节点编辑自由组合)。这说明你的产品直觉是"让用户更自由"。这是好的产品方向,但需要警惕"自由度过载" — 普通用户要的不是无限自由,是"对的限制下做出更好的作品"。
- 你最初说"节点编辑",我误以为是 ComfyUI 那种技术派。澄清后才明白你说的是即梦 AI Studio / Martini 那种"画布 + 预定义节点"工作台。这个澄清非常关键 — 它把"节点编辑"从"不必要的高风险投入"变成了"产品的终极形态"。**说明你脑子里有清晰的目标,但表达时压缩了**,这是创始人常见的沟通模式,以后展开 plan 时要慢一点。
- 你说"看下当前主流的生成路线" — 这说明你愿意被市场调研改变想法,而不是固执于自己的直觉。这是好的特质。即梦 2026 日营收 7000 万 RMB 这个事实应该让你警觉: 这个赛道已经有强对手,你的节点编辑必须有差异化护城河 (PixelVault 的 BYOK 模式 + 用户自己的 LoRA + 私有素材库,是即梦给不了的)。
