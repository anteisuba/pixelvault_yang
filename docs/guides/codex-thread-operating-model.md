# Codex 四线程工作模式

## 目的

为项目建立固定的多线程协作结构，把规范、探索、执行拆开，避免所有事情都混在一个 thread 里。

固定线程名称：

- `规范`
- `探索`
- `前端`
- `后端`

这四个 thread 应该在项目 UI 中长期置顶，并保持名称稳定。

## 线程职责

| Thread | 主要职责                              | 主要产出                             | 不应该漂移成         |
| ------ | ------------------------------------- | ------------------------------------ | -------------------- |
| `规范` | 沉淀项目规范、法则、AI 元编码         | `docs/guides/*.md`、`AGENTS.md` 索引 | 日常功能开发线程     |
| `探索` | 理解代码、调研外部仓库、拆解大任务    | `docs/plans/*.md`                    | 长线实现线程         |
| `前端` | 落地 UI、交互、布局、i18n 相关实现    | 代码变更、必要验证                   | 无计划的临时改动线程 |
| `后端` | 落地服务、API、鉴权、存储、数据库实现 | 代码变更、必要验证                   | 无计划的临时改动线程 |

## `规范` Thread 规则

- 只写长期有效的规范，不写一次性任务笔记。
- 规范文档统一放入 `docs/guides/`。
- 新增或修改 guide 后，都必须同步检查并更新 `AGENTS.md` 中的 guide 索引描述。
- 当 `前端` 或 `后端` thread 在实现中发现反复出现的约束、坑点、模式时，应回流到这里沉淀。

### `规范` Thread 的准入判断

- 在动手写 guide 前，先判断候选内容是否具有长期治理价值。
- 只有这类内容适合进入规范层：
  - 会长期约束仓库协作方式的规则
  - 可重复执行的工作流
  - 会反复影响 AI / 人类协作质量的边界与纪律
- 下列内容应明确排除，不沉淀为 guide：
  - 一次性任务笔记
  - 临时方案
  - 单次 bug 处理结论
  - 只服务于当前任务拆解的计划
- 当内容不具备长期治理价值时，`规范` thread 应明确拒绝把它上升为规则，并建议转交 `探索` thread 处理。

### `规范` Thread 的沉淀方式

- 优先补丁式更新现有 guide，避免为局部补充重复开新主题。
- 只有当现有 guide 无法承载该主题时，才新增 guide。
- 不把临时决定、实现细节或单次争议直接提升为长期规则；需要先确认它已经表现为稳定模式或长期约束。

### `规范` Thread 的输出纪律

- 每次输出都应明确说明：
  - 本次沉淀的长期规则是什么
  - 为什么它值得进入规范层
  - 修改了哪些 guide / 索引
  - 哪些内容被明确排除，不属于长期规范

## `探索` Thread 规则

- 专门用于看代码、看架构、看外部参考、拆任务。
- 任务拆解结果统一放入 `docs/plans/`。
- 大任务尽量先形成计划文档，再进入执行线程。
- 如果执行中发现计划已经失真，应更新计划文档，而不是让实现悄悄偏离。

建议一个计划文档至少包含：

- 背景
- 现状
- 目标与非目标
- 约束
- 拆解步骤
- 风险
- 验证方式

## `前端` / `后端` Thread 规则

- 这两个 thread 是执行层，职责是写代码，不是堆积探索过程。
- 每一次代码修改都必须使用 plan 模式。
- 如果改动同时涉及前后端，任选一个执行 thread 即可，但计划里要明确写出涉及的层。
- 实现后如果产生了新的稳定约束，应回流到 `规范`。
- 实现后如果改变了任务边界、依赖顺序或拆解策略，应回流到 `探索`。

## 推荐工作回路

1. `规范` 沉淀长期法则
2. `探索` 把问题理解清楚并写成计划
3. `前端` 或 `后端` 在 plan 模式下执行
4. 执行中产生的新认知回流到 `规范` 和 `探索`

这套回路的目标不是增加流程，而是让项目随着理解加深而变得更稳定、更可预测。
