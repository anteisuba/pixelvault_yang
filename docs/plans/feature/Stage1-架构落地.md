### 1.2 SourceImageEntry schema 扩展 (`src/types/index.ts`)

```ts
export const SourceImageEntrySchema = z.object({
  // ─── 稳定标识 (P0 新增, 解决 Codex 指出的 assetIndex 债务) ─
  id: z.string().uuid(), // 必填 uuid, Stage 2/3 用这个引用
  url: z.string().min(1),
  viewType: z.enum(CHARACTER_CARD.VIEW_TYPES).default('other'),
  label: z.string().max(60).optional(),
  // ─── 新增字段 (向后兼容: 全部 optional) ─────────────
  prompt: z.string().max(2000).optional(), // 生成此资产时的 prompt
  sourceModel: z.string().max(80).optional(), // 'gpt-image-2' | 'gemini-3-pro-image-preview' | 'user-upload'
  createdAt: z.string().datetime().optional(), // ISO 8601, 生成时刻
  generationId: z.string().optional(), // 关联 Generation 表的 id (复用现有计费/历史)
})

export type SourceImageEntry = z.infer<typeof SourceImageEntrySchema>

/**
 * Asset reference protocol — Stage 2/3 使用这个引用具体资产,
 * 永远不要保存 assetIndex (会在删除/排序后变成坏引用).
 */
export const CharacterAssetRefSchema = z.object({
  characterCardId: z.string().uuid(),
  assetId: z.string().uuid(), // 对应 SourceImageEntry.id
})
export type CharacterAssetRef = z.infer<typeof CharacterAssetRefSchema>
```

**关键变化**:

- `id` 字段**必填** uuid (Codex 指出的 P0 修正)
- 新增 `CharacterAssetRef` 引用协议, Stage 2 VideoScript / Stage 3 节点画布只能用这个引用
- 现有 sourceImageEntries 历史数据 (无 id 字段) 需要一次性数据迁移补 uuid (零停机, 见 §1.4)

### 1.4 历史数据迁移 (一次性 patch)

由于 `id` 改为必填, 现有 CharacterCard.sourceImageEntries 必须补 uuid:

```ts
// scripts/migrate-source-image-entries-add-id.ts
// 运行一次, 给所有现有 sourceImageEntries 补 uuid
import { db } from '@/lib/db'
import { v4 as uuidv4 } from 'uuid'

const cards = await db.characterCard.findMany({
  where: { sourceImageEntries: { not: null as never } },
})

for (const card of cards) {
  const entries = card.sourceImageEntries as unknown as Array<
    Record<string, unknown>
  >
  const patched = entries.map((e) => ('id' in e ? e : { id: uuidv4(), ...e }))
  await db.characterCard.update({
    where: { id: card.id },
    data: { sourceImageEntries: patched as never },
  })
}
```

这是 Stage 1 Week 0 任务 (在 schema 改之前跑).

export async function generateThreeViewSet(
  clerkId: string,
  cardId: string,
  options: { sourceModel: 'gemini' | 'gpt-image' },
): Promise<SourceImageEntry[]> {
  const card = await getCharacterCardById(clerkId, cardId)
  const primaryImage = card.sourceImageUrl
  const characterPrompt = card.characterPrompt

  // 三视图 prompts
  const viewPrompts: Array<{ viewType: SourceImageViewType; prompt: string }> = [
    { viewType: 'front', prompt: `${characterPrompt}, front view, full body, T-pose, neutral expression, simple background` },
    { viewType: 'side', prompt: `${characterPrompt}, side view (90 degrees), full body, neutral pose, simple background` },
    { viewType: 'back', prompt: `${characterPrompt}, back view, full body, simple background` },
  ]

  const modelId = options.sourceModel === 'gemini'
    ? AI_MODELS.GEMINI_PRO_IMAGE
    : AI_MODELS.OPENAI_GPT_IMAGE_2

  // 并行生成 3 张, 每张都用主图作为 reference 锁定身份
  const results = await Promise.allSettled(
    viewPrompts.map(({ prompt, viewType }) =>
      generateImageForUser(clerkId, {
        prompt,
        modelId,
        aspectRatio: '1:1',
        referenceImages: [primaryImage],  // 单图 reference, Gemini 接受最多 14 张
      }),
    ),
  )

  // 成功的入 sourceImageEntries
  return results
    .map((result, idx) => /* 映射为 SourceImageEntry */)
    .filter(Boolean) as SourceImageEntry[]
}

**用户决策**: GPT-Image 2 作为默认 Provider (基于实际使用反馈, 效果优于 Gemini)

// 现在: const effectiveRefImage = referenceImages?.[0] ?? referenceImage

const allRefs = referenceImages?.length
  ? referenceImages
  : referenceImage
    ? [referenceImage]
    : []

if (allRefs.length > 0) {
  // 用 /v1/images/edits endpoint (支持多 image 输入)
  const formData = new FormData()
  formData.append('model', getExecutionModelId(modelId))
  formData.append('prompt', prompt)
  formData.append('size', size)

  for (const ref of allRefs) {

    const extension = mimeType.split('/')[1] ?? 'png'
    formData.append(

      `reference.${extension}`,

### 4.3 信息架构原则

1. **Tab 分组 by viewType** — 用户的心智是"这个角色的 X 类型资产"
2. **每个 tab 有"一键生成"** — 不要让用户每次手动操作
3. **每个资产 hover 显示** — sourceModel / prompt / createdAt
4. **资产可以单独删除 / 替换 / 编辑 label**
5. **资产可以"复制到 Studio 作为 reference"** — 一键带回主创作流

## 9. Performance 评估

- **N+1 风险**: 无. Stage 1 资产都通过 CharacterCard 一次取出 (sourceImageEntries 是单字段 Json)
- **多图 reference 性能**: Gemini Pro 单次调用接受多图, 但每张 ref 增加 token 消耗 (~2k tokens per image). 三视图任务用 1 张 ref, 服装任务用 2 张 ref, 都在 reasonable 范围
- **并行调用**: 三视图 / 表情批量用 `Promise.allSettled`, 已是最佳实践
- **JSON 字段读写**: CharacterCard.sourceImageEntries 在用户更新角色时整体读写. 单角色资产数 <50 应该没问题. 如果超过会触发 PG TOAST, 但 Stage 1 不会到这量级
- **图片传输**: GPT-Image / Gemini Pro 接受 base64 或 URL. 用 R2 URL 避免重复 upload (sourceImageUrl 已经是 R2 URL)
- **生成时长**: GPT-Image 单张 ~30-60s, Gemini Pro ~10-20s. 三视图并行 = 总时长以最慢一张为准

### Performance 关键决策

- 三视图 batch 是否要排队 / 限流? **不要**. 让 Promise.allSettled + 上游 RATE_LIMIT_CONFIGS.imageTransform 兜底
- 表情批量 4-8 张同时跑会不会撞 Provider QPS? Gemini Pro 个人 API key 通常 5 QPM. **必须串行 + 进度展示**, 不要并行 8 个 (用户体验差且会撞限流)

## 10. Failure modes & 错误处理

| 失败场景                  | 检测                          | 用户看到                                  | 是否有测试 |
| ------------------------- | ----------------------------- | ----------------------------------------- | ---------- |
| Provider API key 失效     | generateImageForUser throw    | "请检查 API key" toast + 跳 ApiKeyManager | GAP        |
| Provider timeout (60s)    | AbortSignal.timeout           | "生成超时, 请重试" + 失败格子 Retry       | GAP        |
| 三视图 3/3 全失败         | Promise.allSettled all reject | "全部生成失败" + 提示换 model             | GAP        |
| 三视图 2/3 部分失败       | mixed                         | 2 张成功显示 + 1 个失败格子 Retry         | GAP        |
| Card 在另一 tab 被删      | DB record not found           | "角色不存在或已删除" + 跳列表             | GAP        |
| sourceImageEntries 超 100 | App-level check               | "资产数量已达上限, 请删除部分"            | GAP        |
| 服装 ref 图过大           | upload preflight              | "图片需 <10MB"                            | GAP        |
