## 1. 产品定位

Canvas 是 PixelVault 的北极星能力之一（与 LoRA 并列双核，见 [`../product.md`](../product.md)）。它是**长视频导演台**：把"剧本 → 分镜 → 逐镜生成 → 拼接"的可控流程放在一张无限画布上编排，让用户复用资产、跨模态绑定、维持角色一致性、组织镜头序列和长流程。

Canvas 的核心承诺是：

- **可控编排**：不是"一句 prompt 出一个结果"，而是把创作拆成可见、可复用、可追溯的节点与关系。
- **一致性**：角色、声音、场景、镜头规格通过卡片 + 参考约束的组合跨镜头保持稳定，不靠单一 prompt。
- **跨能力汇聚**：图片、声音、参考视频、剧本文本作为素材汇入视频生成，画布负责它们如何被组织、绑定和送进生成请求。

与 Studio 的分工是产品级契约：**Studio（Image/Video/Audio）= 轻量、单次、快速得到一个结果的入口；Canvas = 高级编排与连续制作**。同一能力两处都在，不是重复，是深浅两档。

## 2. Canvas 域负责什么

- 在无限画布上创建、排布、连接和编辑创作对象（节点），维持画布视口、平移缩放、选择和布局。
- 承载导演工作流：剧本大纲 → 镜头 → 逐镜生成镜头图 → 视频镜头 → 片段合并成长片。
- 组织跨模态素材的汇聚与绑定：角色/场景/镜头图/关键帧/音色/参考视频/剧本文本如何成为一次视频生成请求的成分。
- 维持资产复用：一个身份（角色/场景收集器）在多个镜头间复用；素材在画布上散件或经收集器整体参照。
- 把画布的图结构编译为真实生成请求：收割上游成分、装配 payload、翻译创作名 → provider 位置 token、容量校验、图例注入。
- 承载画布助手：理解画布上下文、投影剧本为节点、在确认后调用已注册的画布能力。
- 项目管理：画布工作流项目的创建、切换、命名、保存与刷新恢复（与 Assets 的 Project 归类文件夹是两回事）。
- 保存生成结果的谱系（lineage）与必要参数，使画布产出可追溯并进入长期资产体系（Assets）。

## 3. Canvas 域不负责什么

| 相邻域       | 边界                                                                                                                                                                      |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Studio Image | 负责不以编排为中心的通用/单次图片生成与专业图片编辑。Canvas 消费图片作为镜头素材，不复制完整图片工作台；六项图片编辑能力在画布内是"图片对象能力"，主力编辑仍可留 Studio。 |
| Studio Video | 负责轻量短片快速入口。Canvas 承接长视频/系列镜头/角色一致性/分镜/参考约束/片段合并；两边不合成一套拥挤表单。                                                              |
| Studio Audio | 负责主力音频生成（TTS/试音/音效/音乐）。Canvas 只消费音频资产与音色身份作为视频成分，不把完整 Audio 页搬进画布。                                                          |
| LoRA         | 负责 LoRA 发现/挂载/组合/训练。LoRA 可以成为画布节点/生成的输入，但画布不在工作台内复制 LoRA 编排。                                                                       |
| Assets       | 负责生成结果与素材的长期归档、整理、批量与复用。画布产出资产，但不复制完整资产管理器；画布内的素材复用走既有 AssetSelector 入口。                                         |
| Cards        | 负责角色/画风/声音/背景卡的持久身份管理。画布消费卡片作为一致性单元并可就地新建/编辑局部，但卡片的长期管理页归 Cards。                                                    |
| Prompts      | 负责持久化的个人配方与版本复用。画布负责本次编排中的装配状态；用户明确保存后才成为 Prompts 域的配方。                                                                     |
| 执行基础设施 | Runner、provider、队列、回调负责真实执行。画布只暴露已接通的能力，不用 UI 伪装未支持的参数。                                                                              |

## 4. 当前信息架构事实（业务事实，非视觉命令）

- 主路由：`/studio/node`（用户可见名"画布"）。
- 画布对象（节点）按模态与角色划分，当前实现的现役类型：图片（散图 / 收集器角色 / 收集器场景 / 镜头图 / 特写 / 关键帧）· 视频生成（seedance）· 参考视频 · 音色 · 片盒（videoMerge）· 剧本文本（shotText，退役中）。`composer`/`agent` 为待删旧 planner。legacy 逐角色图片类型仅兼容旧存档。
- 三个常驻工作面（对齐 Haivis 空间纪律，作为**行为**继承）：画布主区、可收起的助手栏、承载素材复用的卡匣/工具 dock。
- 关系与绑定：数据层为节点 + 边的有向图，合法性由 `node-connection-rules.ts` 定义。**绑定手势与关系呈现是当前实现的交互设计（见 §7 与 §9），底层图模型是锁定业务。**
- 项目状态存 `NodeWorkflowProject.state`（单 Json 列，Zod 层校验）；节点/边/画布外观入其中，viewport 不入存档（会话内保留、刷新回默认）。
- 当前默认视图缩放、节点类型枚举、连接矩阵、收割装配、字段定义以 `src/constants/node-*`、`src/lib/node-*`、`src/types/node-workflow.ts` 与 `StudioNodeWorkbench` 为事实源。

这组结构是当前业务事实，不是未来页面必须使用现有壳层、卡片形态或空间组织的视觉命令。未来可以改变空间结构与交互组织，但不得丢失用户任务、图数据契约、可达性和 URL/项目状态契约。

## 5. 核心工作流

1. **从剧本起手**：跟助手聊大纲 → 助手投影镜头/节点（autospawn）→ 逐镜细化。
2. **手动搭镜**：添加节点（角色/场景/镜头图/图像/参考视频/音色/视频生成/片盒）→ 组织素材与身份。
3. **一组素材 → 一个视频**（最高频导演任务，倒推自 Seedance 袋型合同）：把一组素材（角色出场图组 + 场景 + 音色 + 参考视频 + 镜头文本）汇入一个视频生成节点 → 写提示词 → 生成。
4. **一致性复用**：一个收集器卡（角色/场景，含出场图组 + 绑定音色）在多个镜头/视频间整体参照，维持跨镜身份一致。
5. **静帧先审再喂**：镜头图卡作为中间产物，先产静帧审核，再喂视频。
6. **拼接长片**：多个视频片段进片盒，保序重排 → 合成长片。

未来 UI 设计必须先证明这些工作流如何更清楚、更有辨识度，再讨论造型；不能只为当前组件换皮。

## 6. 领域词汇

| 术语                  | 在 PixelVault 中的含义                                                                                  |
| --------------------- | ------------------------------------------------------------------------------------------------------- |
| 节点                  | 画布上的一个创作对象（图片/视频/音色/收集器/片盒/剧本文本等）；有类型、状态、媒体和参数。               |
| 收集器卡（角色/场景） | 一个身份的整体档案：名字 + 图集（可标"出场"）+ 绑定音色；被镜头/视频整体参照，是一致性复用的单位。      |
| 出场组                | 收集器卡贡献给一次生成的有序图组（主图恒第一 + 标了"出场"的额外图）；可按单个镜头覆写。                 |
| 散图                  | 画布上一张无名的图片素材；可直接作参考，被吃后折叠为成分。                                              |
| 镜头图卡              | 中间产物：吃角色/场景 → 产静帧，先审再喂视频。                                                          |
| 视频生成节点          | 汇聚一组成分 + 提示词 → 产视频片段（当前实现名 seedance，非长期产品类型名）。                           |
| 片盒                  | 保序装订成片段的容器：吃视频成片/参考视频段/别盒长片 → 合成长片；判据=占时间线的画面段。                |
| 音色                  | 声音身份（可绑角色成为听觉身份），作视频成分；不产成品对白。                                            |
| 绑定 / 收割           | 把上游节点的素材组织进一次生成请求的过程；收割 = 沿边解析上游成分并装配 payload。                       |
| 关系呈现              | 节点间供给/制片关系如何对用户可见（当前实现为两级：制片流常显、供给按需——**是交互设计，非锁定业务**）。 |
| 卡匣                  | 收集器卡的复用注册表（镜像托盘）；一卡多镜从这里再喂。                                                  |
| 画布助手              | 理解画布上下文、投影剧本、在确认后调用画布能力的对话式脑；不是通用聊天。                                |

## 7. 不能破坏的业务约束（视觉重设计必须保住）

- **数据层图模型不动**：节点 + 边 + 收割装配 + autospawn + 存档解析 + payload 共享装配（`assembleReferenceImagePayload` 等）是锁定业务；任何视觉/交互重设计只能改呈现，不能切断建图、收割、翻译、容量校验、图例注入这条链。
- **provider 契约保真**：Seedance 2.0 Reference 的袋型合同（1 prompt + image_urls/audio_urls/video_urls）、上限（image 9）、@token 翻译、音频绑定名——UI 重设计不得伪装或绕过真实 provider 能力。
- **一致性单位不退化**：收集器卡"名字 + 出场图组 + 音色三件套整体参照"、一卡多镜复用、每镜覆写——是画布相对 Studio 的核心价值，不能退化成散图堆。
- **深浅两档分工成立**：Studio 轻量单次入口继续存在；画布不吞并 Studio，也不把画布降成通用白板。
- **生成结果谱系与资产化**：画布产出保留足够 lineage 且能进 Assets；重构不切断持久化与回放路径。
- **合法性事实源唯一**：`node-connection-rules.ts` 是绑定合法性的唯一事实源；呈现层查表，不反向影响合法性。
- **全局品质底线继承**：可访问性、键盘可达、焦点管理、状态真实性、reduced-motion、ResponsiveOverlay、触屏软键盘、i18n 三语——见 `brand-dna.md` 全局品质底线，画布不例外。

## 9. 尚未拍板（交 Fable 设计阶段）

- Canvas 未来域级视觉身份与设计性格（当前暗炭/纸卡/石绿已退役为规则，不作为默认答案）。
- **关系呈现范式完全开放重议（owner 2026-07-19 明确）**：当前实现"吞噬手势 + 两级关系呈现 + 披露三档 + 卡匣"只是本轮业务收口的一种交互设计，**不是拍死的约束**——Fable 设计阶段**连"要不要还用吞噬、要不要还用两级墨线"都可以重新提**，可提出结构完全不同的绑定手势与关系组织。唯一硬边界 = §7 的业务能力（一组素材如何汇入一次生成、一致性单位、合法性事实源、数据层图模型）必须仍能达成，至于用什么交互抵达完全由设计定。
- 节点卡、卡匣、助手栏、详情面板、视频详情等标志性组件的未来形态。
- 三个结构明显不同的概念方向及其关键切片。
- 与相邻域（尤其 Studio Video/Image、Cards）在视觉辨识度上如何区分，差异来自工作流/结构而非换色。
- 排序：Canvas 视觉重设计相对 Prompts 试点与 Canvas 业务收口的先后。

这些问题必须逐项与 owner 讨论、由 Fable 出设计，不得由 AI 从旧文档或当前页面自行补全。
