## 0. 速查（TL;DR）

- **范围**：剧本 → 分镜图 → 视频 clip → 拼接的完整管线（非纯文本）
- **粒度**：按目标总时长反推场景数（30s / 60s / 120s · 每场景约 5-8s）
- **数据模型**：新建 `VideoScript` + `VideoScriptScene`（与现有 `Story` 完全分离）
- **入口**：Studio `outputType = 'video'` 下的子功能（非独立路由）
- **前置**：先把 Studio video mode 的 UI 对齐 image mode，再叠剧本功能
- **一致性**：Character Card 与首帧 ref 两种模式都支持，用户选
- **Provider**：分镜图走 `recipe-compiler`（兼容 Style Card）· 视频 clip 先给 2 个选项（Seedance 2.0 Fast · Kling Pro）
- **运行模式**：每阶段用户确认（剧本 → 图 → 视频 → 拼接 都有 checkpoint，不一键到底）

## 2. 全部 Rejected Directions（RVS1-RVS8）

| #    | Rejected                       | 拒绝理由                                                                              |
| ---- | ------------------------------ | ------------------------------------------------------------------------------------- |
| RVS1 | A. 纯文本剧本                  | 用户明确要完整视频管线，纯文本导出后仍要手工去生成，价值点不在 PixelVault 的差异化上  |
| RVS2 | B. 剧本 + 分镜图（不做视频）   | 与 A 同理；仍是半成品                                                                 |
| RVS3 | 固定 N 场景档（3 / 5 / 8）     | 用户体感是"我要做 30 秒短片"，而不是"我要 5 个场景"；档位模型与用户心智模型不匹配     |
| RVS4 | LLM 完全自适应决定场景数       | Token 成本不可控 · 输出方差大 · 与视频总时长耦合需要再校正一次 → 复杂                 |
| RVS5 | 复用扩展现有 `Story` 模型      | 会把漫画/滚动叙事的领域模型污染成视频领域 · 字段爆炸 · 后续演进两边都难改             |
| RVS6 | 独立 `/script` 路由            | 违反功能结论书 F19 RF17（避免两个 Studio 让用户困惑）· 视频剧本本质是视频模式的子功能 |
| RVS7 | 仅首帧 ref 或仅 Character Card | 首帧 ref 启动成本低但质量差 · Character Card 质量好但门槛高；二选一会牺牲一类用户     |
| RVS8 | 一键端到端无确认               | 每步成本不低（每场景 =1 张图 + 1 段视频），无确认会让失败场景浪费上游所有成本         |

## 3. 北极星定义

```
主题输入
  ↓ LLM 生成 · 用户确认
结构化剧本（场景列表：镜头/动作/台词/时长/转场）
  ↓ recipe-compiler · 用户确认
分镜图（每场景 1 张，Character Card 或首帧 ref 保一致性）
  ↓ i2v Provider · 用户确认
视频 clip（每场景 1 个，i2v 用分镜图 + scene.action 驱动）
  ↓ 拼接 · 用户确认
成片（MP4，落 R2，进 Gallery）
```

**产品原则**：

1. **完整管线，但每阶段可停** — 用户可以只用剧本，可以只用到分镜图，可以只用到单 clip
2. **每阶段独立扣费** — 不预扣成片成本，按实际推进阶段计费
3. **结构化优先于自由度** — 剧本用固定 schema（非自由文本），便于下游机械化处理

## 4. 数据模型设计（Phase 1 落地）

### 4.1 Prisma 新增

```prisma
model VideoScript {
  id              String   @id @default(cuid())
  userId          String
  user            User     @relation(fields: [userId], references: [id])
  topic           String                 // 用户输入的主题/想法
  targetDuration  Int                    // 30 | 60 | 120 (seconds)
  totalScenes     Int                    // 反推出的场景数
  status          VideoScriptStatus      // DRAFT / SCRIPT_READY / FRAMES_READY / CLIPS_READY / COMPLETED / FAILED
  consistencyMode String                 // 'character_card' | 'first_frame_ref'
  characterCardId String?                // 当 consistencyMode = 'character_card' 时必填
  styleCardId     String?                // 可选 · 全局画风
  videoModelId    String                 // 'seedance-2-fast' | 'kling-pro' 二选一
  finalVideoUrl   String?                // 拼接完成的 R2 URL
  scenes          VideoScriptScene[]
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
}

model VideoScriptScene {
  id              String   @id @default(cuid())
  scriptId        String
  script          VideoScript @relation(fields: [scriptId], references: [id], onDelete: Cascade)
  orderIndex      Int                    // 0-based 场景序号
  duration        Int                    // 此场景时长（秒）
  cameraShot      String                 // 'close-up' | 'medium' | 'wide' | 'establishing' ...
  action          String                 // 场景动作描述（LLM 生成，驱动 i2v motion）
  dialogue        String?                // 台词（可选）
  transition      String                 // 'cut' | 'fade' | 'dissolve'（Phase 4 仅实现 'cut'）
  frameGenerationId String?              // 分镜图 Generation id
  clipGenerationId  String?              // 视频 clip Generation id
  status          VideoScriptSceneStatus // PENDING / FRAME_GENERATING / FRAME_READY / CLIP_GENERATING / CLIP_READY / FAILED
  errorMessage    String?
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
  @@unique([scriptId, orderIndex])
}

enum VideoScriptStatus {
  DRAFT
  SCRIPT_READY
  FRAMES_READY
  CLIPS_READY
  COMPLETED
  FAILED
}

enum VideoScriptSceneStatus {
  PENDING
  FRAME_GENERATING
  FRAME_READY
  CLIP_GENERATING
  CLIP_READY
  FAILED
}
```

### 4.2 场景数反推规则

```ts
// src/constants/video-script.ts (Phase 1 新增)
export const VIDEO_SCRIPT_TARGET_DURATIONS = [30, 60, 120] as const
export const SCENE_DURATION_RANGE = { min: 5, max: 8 } as const

export function deriveSceneCount(targetDuration: number): number {
  // 策略：优先让每场景接近 6s（平衡节奏）
  // 30s → 5 场景 · 60s → 10 场景 · 120s → 20 场景
  const avgSceneDuration = 6
  return Math.round(targetDuration / avgSceneDuration)
}
```

## 5. Provider 策略（Phase 2-3 落地）

| Stage         | 路径                                                                     | Provider                           | 成本边界            |
| ------------- | ------------------------------------------------------------------------ | ---------------------------------- | ------------------- |
| **分镜图**    | `recipe-compiler.compileRecipe()` + `generate-image.service`（完全复用） | 任何支持 i2i 的 image provider     | 每场景 1 张         |
| **视频 clip** | `generate-video.service`（复用）· i2v mode · reference = 分镜图          | Seedance 2.0 Fast **或** Kling Pro | 每场景 1 段（5-8s） |
| **拼接**      | Phase 4 决策：WebCodecs（客户端）vs FFmpeg（服务端）                     | TBD                                | 单次                |

**一致性路径分支**（VS6）：

- **Character Card 路径**：所有分镜图调用 `recipe-compiler` 时附 `characterCardId`，自动带 LoRA + prompt tag → 高一致性
- **首帧 ref 路径**：第 1 场景用 t2i 生成，第 2-N 场景用第 1 张作为 reference image → i2i with ref → 中等一致性，启动成本低

## 6. 阶段演进路径

### Phase 0（前置）· Video UI 布局归一

详细工作包见 `03-工作包细分-Phase0.md`（v3：9 个 WP，Effort S×5 / M×4，现已作为历史 Phase 0 规划归档）。

**范围演变**：

- **v1（2026-04-18 上午）**：误判 video 与 image 共组件树 → 拆出 14 处差异 → 废弃
- **v2（2026-04-18 下午）**：归一"视觉语言"（圆角/边框/表面/label/间距 token） → 已合入 PR1-3 → 但用户看到实际布局仍浪费空间（video 锁在 `max-w-3xl` 768px）
- **v3（2026-04-18 晚）**：**布局归一** — video 移除独立 form，全走 `StudioFlowLayout`（canvas + dock + gallery），与 image mode 真正一致

**v3 Phase 0 目标**：

- `StudioWorkspace` 移除 `outputType === 'video'` 短路分支
- `StudioPromptArea` 支持三模式（image / video / audio）
- `StudioFormState` 扩展 video 专属字段（duration / resolution / longVideoMode 等）
- 新 `StudioVideoParams` dock panel 收纳 video 参数
- `StudioToolbarPanels` 按矩阵显隐
- 弃用 `VideoGenerateForm` / `VideoFormSettings`

**不做**（依然）：

- 底层生成逻辑改动（`useGenerateVideo` / adapter / service 全不动）
- 拼接方案预研
- 新增 video 模型 / provider

> v2 视觉 token 归一的工作在 v3 完成后大部分失效（因 VideoGenerateForm 被删），但零损害 — `StudioGallery.handleRemix` 和 `CommandPalette.switchMode` 两处三模式支持在 v3 仍然被需要。

### Phase 1 · VideoScript 后端骨架 + 剧本生成 LLM

- Prisma migration：新增 2 表 + 2 enum
- Types：`src/types/video-script.ts` + Zod schema
- Service：`src/services/video-script.service.ts`（`generateScript(input)` → LLM 结构化输出）
- API：`POST /api/video-script` + `PATCH /api/video-script/[id]` + `GET /api/video-script/[id]`
- Constants：`src/constants/video-script.ts`（时长档 + 反推规则 + LLM system prompt）
- UI：`StudioScriptPanel`（Studio video mode 下）+ `ScriptEditor`（场景可编辑）
- **验收点 1**：用户输入主题 → LLM 生成剧本 → 可编辑 → 确认后 status=SCRIPT_READY

### Phase 2 · 分镜图生成

- Service：`src/services/video-script/generate-scene-frames.ts`（并行调用现有 `generate-image` + `recipe-compiler`）
- 一致性路径分支实现（VS6）
- UI：`SceneFrameGrid`（每场景一格，显示图 + 可 Retry / 切卡）
- API：`POST /api/video-script/[id]/generate-frames`
- **验收点 2**：分镜图全部确认后 status=FRAMES_READY

### Phase 3 · 视频 clip 生成

- Service：`src/services/video-script/generate-scene-clips.ts`（复用 `video-pipeline` clip 状态机）
- 2 Provider 选型生效
- UI：`SceneClipGrid`（每场景播放器 + 单格 Retry）
- API：`POST /api/video-script/[id]/generate-clips`
- **验收点 3**：所有 clip 确认后 status=CLIPS_READY

### Phase 4 · 拼接导出

- 拼接方案选型（单独决策）
- Service：`src/services/video-script/compose-final-video.ts`
- API：`POST /api/video-script/[id]/compose`
- **验收点 4**：成片进 R2 + Gallery · status=COMPLETED

### Optional Phase 5+

- 转场（淡入淡出 / dissolve）
- BGM / 配音（复用 `generate-audio`）
- 字幕烧录
- 剧本市集（用户分享剧本模板）

## 8. 已识别风险与缓解

| #   | 风险                                            | 缓解                                                                               |
| --- | ----------------------------------------------- | ---------------------------------------------------------------------------------- |
| R1  | 分镜图一致性不稳定（首帧 ref 路径）             | VS6 同时保留 Character Card 路径作为"高配"选项，UI 明确标注质量差异                |
| R2  | 视频 clip 生成失败率高 → 部分场景卡死           | Phase 3 复用 `video-pipeline` 的 clip 状态机 → 单格可 Retry，不阻塞其他场景        |
| R3  | N 场景计费爆炸（120s = 20 场景 = 20+20 次生成） | Phase 1 UI 明确显示总成本预估；FREE_TIER 用户限制 targetDuration ≤ 30s             |
| R4  | LLM 结构化输出格式不稳定                        | 复用 `llm-output-validator` + Zod 结构校验 + 多轮 retry + 降级到 template fallback |
| R5  | 拼接方案选型卡死 Phase 4                        | Phase 4 启动前单独开一轮技术决策；Phase 0-3 完全与拼接解耦                         |
| R6  | Studio UI 对齐改动面大，引入回归                | Phase 0 严格不动生成逻辑，只改 UI；工作包逐个验收；保留 feature flag 可回滚        |
