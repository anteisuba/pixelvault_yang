## Context

PixelVault 的视频后端已经完整（12 个模型、短视频 + 长视频 Pipeline、clip chaining），但前端体验停留在"填表"级别。用户选择"影视级视觉叙事"作为愿景，核心差异化是用现有卡片系统（CharacterCard + StyleCard + BackgroundCard）驱动多镜头视频生成，实现 Runway 级的叙事能力但用卡片系统解决一致性问题。

## Problem Statement

当前 Studio 的视频模式是一个 prompt textarea + model selector + 参数面板。用户像在提交 API 请求，不像在创作影视作品。后端 Pipeline（clip chaining、轮询、重试、取消）都能跑，但前端没有将卡片系统和视频生成连接起来。9/12 个视频模型支持 I2V，但 UI 没有利用这个能力。

## What Makes This Cool

**自动多角度覆盖（Coverage）**：每个场景自动生成 4 个镜头角度（远景 establishing / 中景 medium / 特写 close-up / 反应 reaction），这是 AI 视频工具领域没有人做过的事。结合卡片系统的一致性保证，输出不是"AI 生成的视频片段"，而是"影视级场景覆盖"。

**卡片 = 一致性引擎**：所有视频工具的最大痛点是多镜头角色飘移。PixelVault 用 CharacterCard 的参考图在每个镜头自动注入 reference image（I2V），用 StyleCard 统一视觉风格，用 BackgroundCard 保持场景一致。零手动维护。

## Agreed Premises

1. **卡片系统 = 视频一致性引擎** — CharacterCard 参考图 + StyleCard 风格参数自动注入每个镜头
2. **UI 是产品差距** — 后端已完整，需要从"表单"升级到"创作工具"
3. **只支持头部模型** — Kling V3 Pro + Veo 3 + Seedance（2.0 API 开放后升级）
4. **故事板模式优先** — 场景卡片组合，不做复杂时间线编辑器

## Cross-Model Perspective (Codex)

Codex 提出 "World Bible → Film Compiler" 愿景：不只是一个场景一个视频，而是定义 cast + style + location + beat → 自动生成 **coverage**（远景、中景、特写、反应、备选）。这把 PixelVault 从图片仓库变成 **AI 电影的导演 OS**。

关键洞察："用户不是对模型数量兴奋，而是对从可复用创意资产出发的一致性叙事兴奋。"

推荐参考：Wonder Unit 的 Storyboarder（开源故事板工具）作为 UX 隐喻。

## Approaches Considered

### Approach A: Card-to-Video Quick Mode
在现有 Video tab 加卡片选择器，单镜头 I2V 生成。Effort: S, Completeness: 4/10。太简单，没有差异化。

### Approach B: Storyboard Mode with Coverage（选定）
故事板 + 自动多角度覆盖。Effort: L, Completeness: 8/10。核心差异化体验，开发复杂度可控。

### Approach C: Director OS
完整 World Bible + LLM 剧本分解 + 连续性引擎。Effort: XL, Completeness: 10/10。Approach B 的自然演进。

## Recommended Approach — Storyboard Mode with Coverage

### 核心概念

```
Story（故事）
  └── Scene Beat（场景节拍）×N
        ├── CharacterCard（角色）
        ├── BackgroundCard（场景）
        ├── StyleCard（风格）
        ├── Action Description（动作描述）
        ├── Camera Direction（镜头指令）
        └── Coverage Shots（覆盖镜头）×4
              ├── Establishing Shot（远景）
              ├── Medium Shot（中景）
              ├── Close-up（特写）
              └── Reaction Shot（反应镜头）
```

### 数据模型变更

> 注意：已有 `Story` 模型用于 StoryPanel 漫画功能，视频故事板使用独立模型避免冲突。

**新增 Prisma Model: `VideoStoryboard`**
```
VideoStoryboard {
  id, userId, title, description
  styleCardId? → StyleCard（全局风格）
  status: DRAFT | GENERATING | PAUSED | COMPLETED | FAILED
  completedBeats: Int（已完成的 beat 数）
  totalBeats: Int
  estimatedCost: Int?（预估积分消耗）
  createdAt, updatedAt
}
```

**新增 Prisma Model: `SceneBeat`**
```
SceneBeat {
  id, storyboardId → VideoStoryboard
  beatIndex: Int（顺序）
  characterCardId? → CharacterCard
  backgroundCardId? → BackgroundCard
  actionDescription: String（用户写的动作描述）
  cameraDirection: String?（运镜指令：pan left, zoom in, etc.）
  modelId: String（选择的视频模型）
  status: DRAFT | GENERATING | COMPLETED | FAILED
}
```

**新增 Prisma Model: `CoverageShot`**
```
CoverageShot {
  id, sceneBeatId → SceneBeat
  shotType: ESTABLISHING | MEDIUM | CLOSEUP | REACTION
  compiledPrompt: String（系统编译的最终 prompt）
  referenceImageUrl: String?（从 CharacterCard/BackgroundCard 提取的参考图）
  generationJobId? → 对应的视频生成任务
  videoUrl: String?
  storageKey: String?
  durationSec: Float?
  status: PENDING | GENERATING | COMPLETED | FAILED | LOCKED
  lockedAt: DateTime?（用户满意后锁定）
}
```

**状态机规则：**
- VideoStoryboard.status = COMPLETED 当且仅当：每个 Beat 至少有 1 个 LOCKED 或 COMPLETED 的 shot
- VideoStoryboard.status = FAILED 当且仅当：任意 Beat 的全部 4 个 shot 均 FAILED 且用户放弃重试
- 部分失败场景：个别 shot FAILED 不影响 Storyboard 整体状态，用户可重试或跳过

**积分模型：**
- 每个 CoverageShot 消耗 1 次视频生成积分（按模型 cost 计费）
- 5 个 Beat × 4 shots = 20 次生成，约 80-100 积分（按 cost 4-5 计算）
- 重新生成额外计费
- **生成前显示预估消耗**，用户确认后开始
- **积分不足处理**：每个 shot 提交前检查余额。余额不足时暂停 Storyboard（PAUSED），已完成的 shot 保留，用户充值后可继续
- 建议：每个 Beat 的 4 个 shot type 可以由用户选择性启用（默认全开，可关闭不需要的 shot type，如纯环境场景不需要 reaction shot）
```

### Prompt 编译器（核心逻辑）

每个 CoverageShot 的 prompt 由系统自动编译：

```typescript
function compileShotPrompt(beat: SceneBeat, shotType: ShotType): CompiledPrompt {
  const character = beat.characterCard  // sourceImageUrl + sourceImageEntries (JSON with viewType)
  const background = beat.backgroundCard // sourceImageUrl + description
  const style = beat.storyboard.styleCard // stylePrompt 字段

  // 从 CharacterCard.sourceImageEntries 中提取最佳参考图
  const entries = character.sourceImageEntries as SourceImageEntry[]
  const portraitEntry = entries.find(e => e.viewType === 'portrait') || entries[0]
  const fullBodyEntry = entries.find(e => e.viewType === 'full_body') || entries[0]

  const shotFraming = {
    ESTABLISHING: "wide shot, full environment visible, character small in frame",
    MEDIUM: "medium shot, waist-up, character centered",
    CLOSEUP: "close-up, face and upper body, emotional detail",
    REACTION: "over-the-shoulder or reaction shot, subtle expression change"
  }

  // 选择参考图：特写用肖像，其他用全身
  const referenceImage = shotType === 'CLOSEUP'
    ? (portraitEntry?.imageUrl || character.sourceImageUrl)
    : (fullBodyEntry?.imageUrl || character.sourceImageUrl)

  return {
    prompt: `${character.characterPrompt}. ${beat.actionDescription}. ${shotFraming[shotType]}. ${background.description}. ${style.stylePrompt}. ${beat.cameraDirection || ''}`,
    referenceImage,
    // StyleCard 没有独立 negativePrompt 字段，使用模型默认值
    negativePrompt: undefined,
  }
}
```

### 视频模型约束

只开放头部模型，按能力推荐：

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 需要长镜头/运镜 | Kling V3 Pro | native_extend, 180s, 物理准确 |
| 需要唇形同步/对话 | Veo 3 (VEO_3) | 最佳唇形同步 |
| 需要快速迭代 | Seedance Pro (fal.ai, cost 4) | I2V + 速度快 + 成本��� |
| I2V 强一致性 | Kling V3 Pro | I2V + native_extend 组���最稳 |

**I2V 兼容性：**
- Kling V3 Pro: ✅ i2vModelId 已配
- Veo 3 (VEO_3): ✅ i2vModelId 已配
- Seedance Pro (fal.ai): ✅ i2vModelId 已配
- Seedance 1.5 Pro (Volcengine): ❌ 无 i2vModelId → **T2V fallback**，prompt 中加强角色描述代替参考图
- Seedance 1.0 Pro (Volcengine): ❌ 无 i2vModelId → **T2V fallback**

### UI 设计

**Studio 新增 "Storyboard" 模式**（与现有 Image / Video 模式平级）

#### 1. Story Overview 页面
- 标题输入（大字，Space Grotesk）
- 全局 StyleCard 标签 + 统计摘要（N beats / ~N shots / est. N credits）
- Scene Beat 列表（**默认折叠，点击展开**）
- "Add Scene Beat" 按钮
- 底部固定：积分预估 + "Generate Story" 按钮
- 最底部：Preview 缩略图条

#### 2. Scene Beat 卡片（折叠/展开）

**折叠态（默认）— 一行摘要：**
- 序号圆圈（terracotta 色）
- 角色/场景缩略图方块（2rem，带首字母）
- 动作描述（单行截断 ellipsis）
- 模型标签（Kling V3 / Veo 3）
- 状态标签（Draft / 2/4 / Done）
- 展开箭头

**展开态（点击后）— 编辑 + Coverage：**
左侧编辑区：
- **Cards & Scene**：卡片 chip 选择器（角色卡 + 背景卡 + 添加按钮）
- **Action**：textarea 动作描述
- **Camera + Model**：两个下拉并排

右侧 Coverage Grid（2x2）：
- 每个 shot：缩略图 + 类型标签 + 状态标签
- 已生成的显示深色背景 + 播放图标
- 可单独重新生成或锁定

Wireframe 文件：`~/.gstack/projects/anteisuba-pixelvault_yang/designs/storyboard-wireframe-v2.html`

#### 3. 生成流程
1. 用户填完所有 Scene Beat → 点击 "Generate Story"
2. 系统为每个 Beat 的每个 shot type 编译 prompt
3. 按 Beat 顺序 + Shot 优先级（establishing → medium → closeup → reaction）提交
4. 使用现有 Pipeline 后端处理轮询和进度
5. 实时更新 Coverage 预览网格
6. 用户可以锁定满意的 shot，重新生成不满意的
7. 全部完成后，自动组接成最终视频

#### 4. Sequential Preview（简化时间线）
- 底部横向滚动条
- 显示所有最终选定的 shot 按顺序排列
- 每个 shot 显示缩略图 + 时长
- 点击播放从任意位置开始预览
- 不支持拖拽重排（后期可加）

### 并发与队列策略

- 每个用户同时最多 **3 个视频生成任务**（复用现有 rate limit 机制）
- Storyboard 生成时，按 Beat 顺序 + 每个 Beat 内并行 4 个 shot
- 每个 Beat 最多并行 2 个 shot（避免触发 provider rate limit）
- 用户离开页面后生成继续（后端 Pipeline 不依赖前端连接）
- 提供 **通知机制**：生成完成后通过 UI notification 提醒用户

### 视频输出策略（MVP 决策）

- **MVP 使用播放列表模式**：不做服务端视频合并（Vercel serverless 60s 超时限制）
- 前端 SequentialPreview 组件按顺序播放各 shot 的视频 URL
- **后期可选**：接入 ffmpeg lambda 或外部服务做真正合并

## Success Criteria

- 用户可以从 CharacterCard + BackgroundCard + StyleCard 出发，定义 3+ 个场景，一键生成带有 4-shot coverage 的多镜头视频
- 角色在不同镜头间保持视觉一致（通过 I2V reference image 注入）
- 单个不满意的 shot 可以单独重新生成，不影响其他 shot
- 生成过程有清晰的进度展示（哪个 Beat 的哪个 shot 在生成）
- 最终输出可以在 Gallery 中展示和分享
