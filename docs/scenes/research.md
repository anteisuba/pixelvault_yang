# Scene · 调查 / 可行性 / 技术选型（research.md）

> 覆盖："XX 能不能做"、"A 方案还是 B 方案"、竞品调研、技术预研（spike）。产出是**拍板依据**，不是代码。适合 Fable 档执行（调查+顶层设计分工）。

## 专属 5 问（开工硬门）

1. **核心问题一句话？判据是什么？**——什么样的结论算"可行"/"选 A"（成本上限、延迟要求、能力底线），先定判据再调查，避免调完还是拍不了板。
2. **时间盒与深度？**——快扫（<1h，结论±方向）/ 深调研（官方文档+竞品实测）/ 最小原型验证（spike 代码）。
3. **约束前提？**——预算（如 RunPod 月 $2-5 硬上限这类）、技术栈边界、直连优先原则、合规红线。
4. **输出形态？**——对话结论+图示 / 调研文档（`docs/plans/xxx-research-YYYY-MM.md`）/ 可跑原型。非 trivial 默认出文档。
5. **拍板后去向？**——转 task packet 执行 / 存档备用 / 判死。调研文档完成使命后按文档生命周期归档。

## 本场景工作流

1. 问 5 问（判据没定就先和 owner 对齐判据）。
2. **官方一手资料**（WORKFLOW 联网核验优先级：官方 API 文档 > SDK > model card > 公告）；动向类可用本地 OpenCLI（twitter/hf/arxiv/reddit 适配器）。
3. 竞品实测（需要时）：claude-in-chrome 实际走一遍竞品流程，截图留证。
4. 最小原型（需要时）：**spike 代码是丢弃式的**——放 scratchpad 或独立分支，绝不合入 main。
5. 产出：现状 → 可选方案对比（成本/能力/风险三列）→ **推荐方向和理由** → 待 owner 拍板的问题（选择框形式）。
6. owner 拍板 → 沉淀：转 `templates/task-packet.md` 交执行，或调研文档归档。

## 必读清单

`references/product.md`（判断是否偏离双核北极星）· 相关 `references/domains/<域>.md` · 相关在飞任务包（别重复调研，先查 `archive/reviews/` 和 memory 是否已调研过）

## 禁改范围默认值

**不改任何产品代码**（调查中发现的 bug 立 task 不顺手修）· spike 代码不进 main · 不因"调查顺利"跳过拍板直接开工。

## 验证标准

结论可拍板（判据逐条有答案）· 关键事实有官方出处链接 · 方案对比含成本与风险 · 推荐明确不骑墙。
