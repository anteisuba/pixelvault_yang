## 0. 核心洞察（决定这片是大是小）

**道具卡 / 自定义卡 ≈ 角色卡 / 背景卡**：三者都是「收集参考图 + 命名 + 被镜头/视频当参考吃」的**收集器卡**，走**同一套现成机制**：

- 收割：`buildReferenceAssetLegendEntries`（S5c/S5d 已有）——参考图集 → image_urls；
- 图例：S5d「分类进图例」——`图N = <名字>（<类型>）` 注入 prompt，让视频 API 理解素材用途；
- 连线：与角色/背景同族，`→ 镜头图卡 / 视频卡`。

所以**不需要发明新注入管线**。S5e 主要是**数据模型 additive**（新增几个 role）+ CAST_SECTIONS 表项 + 图例类型标注 + 连线矩阵扩项。

唯一需要单独想清楚的是**画风卡与全局 StyleCard 的关系**（见 §3）。

## 1. 三类卡规格

| 卡类型       | 收集什么                  | 注入（图例标注）                          | 连线合法性      | 备注                 |
| ------------ | ------------------------- | ----------------------------------------- | --------------- | -------------------- |
| **画风卡**   | 风格参考图集 + 风格描述词 | `图N = <名字>（画风）`；风格词并入 prompt | → 镜头图 / 视频 | 见 §3 StyleCard 关系 |
| **道具卡**   | 道具/物件参考图集         | `图N = <名字>（道具）`                    | → 镜头图 / 视频 | 与角色/背景同构      |
| **自定义卡** | 参考图集                  | `图N = <名字>（<自定义类型名>）`          | → 镜头图 / 视频 | 用户自拟类型名       |

三者都**不吃别的卡**（叶子收集器，只被镜头/视频吃），与角色/背景一致（角色额外吃音色/特写是历史特例，新类型不继承）。

## 2. 数据模型改动（additive，向后兼容）

1. `NODE_IMAGE_ROLE_IDS` 增 `style` / `prop` / `custom` 三个 role（现有 character/background/shot/frame/closeup 之后追加；旧存档 parse 不破坏）。
2. 自定义卡的类型名：节点数据加可选 `customCardLabel: string`（复用 S5d 已加的 `customLabel` 字段？——**待确认：复用还是新字段**，见 §5 决策点）。
3. `node-connection-rules.ts`：`isCharacterOrBackgroundSource` 泛化为 `isReferenceCardSource`（纳入 style/prop/custom），使三者可连入 shot/video；**方向单向**（卡→镜头/视频），不引入新反向边。
4. `node-workflow-prompt.ts`：图例类型标签映射加三项（画风/道具/自定义[取 customCardLabel]）。

## 3. 画风卡 ↔ 全局 StyleCard（关键决策）

全局 StyleCard（product.md）= 结构化风格定义 + 参考样张 + 模型适配 + 参数快照 + 结果反馈 + 轻度/严格两档强度。这是个**重概念**。

**MVP 推荐（方案 A）**：画布画风卡 = **轻量参考收集器**——风格参考图集 + 风格描述词（+ 可选「轻度/严格」一档强度写进图例措辞）。**不接全局 StyleCard 结构**。理由：画布收割走 image_urls + 图例，StyleCard 的结构化参数在画布视频链路里无落点（视频模型吃参考图不吃风格参数快照）；重接成本高、收益低。

**未来（不在 S5e）**：从全局 StyleCard「导入」成画布画风卡（把 StyleCard 的样张 + 描述灌进画风卡），作为资产复用桥。留 backlog。

> 待 owner 确认：画风卡 MVP 走方案 A（轻量收集器），还是必须首发就接全局 StyleCard 结构（方案 B，工程量大得多）。

## 4. 卡匣 / UI 改动

1. `CAST_SECTIONS` 增 3 表项（画风/道具/自定义）——施工图 §6.0「分区由类型清单驱动」已为此留扩展性，加表项即出分区。**自定义卡**：MVP 全部归一个「自定义」分区（不按 customCardLabel 拆多分区）。
2. 匣尾「＋新建」类型选单：角色/场景/画风/道具/自定义（5 项）；选「自定义」时先让用户输类型名。
3. 收集器档案卡面（S5c 的 IdentityCollectorCard）复用——画风/道具/自定义都是「图集 + 名字 + 词条」，与角色/背景同一套卡面（角色的音色区、背景的环境音区对新类型隐藏）。
4. 空分区不占位（§6.0 修正①既有规则）。

## 5. 待 owner 拍板的决策点

1. **画风卡 MVP**：方案 A 轻量收集器（推荐）vs 方案 B 首发接全局 StyleCard 结构。
2. **自定义卡字段**：复用 S5d 的 `customLabel`（图片分类自定义标签）还是新加 `customCardLabel`（语义更清但多一字段）。
3. **画风卡强度档**：MVP 是否带「轻度/严格」一档（影响图例措辞），还是纯参考图先上。
4. **自定义卡分区**：MVP 归一个「自定义」分区（推荐）vs 按类型名动态拆分区（复杂）。
