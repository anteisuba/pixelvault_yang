# LoRA 域 — 还原与定制工作台

> 状态：**active / 稳定业务域契约**（2026-07-19）。本文定义 LoRA 长期负责什么、与相邻域如何分工，以及未来 UI 设计必须保留的业务事实；**不定义字体、颜色、材质、圆角、卡片、布局或动效答案**。
>
> 当前业务施工与回归依据见 [`../pages/lora-workbench.md`](../pages/lora-workbench.md)。未来视觉设计仍须按 [`../../scenes/ui-page.md`](../../scenes/ui-page.md) 完成三个结构方向、关键切片与 owner 确认，不能直接从当前页面或历史方案生成。

## 1. 产品定位

LoRA 是 PixelVault 的北极星能力之一。它不是 Image Studio 的一个参数抽屉，而是围绕模型能力形成的独立创作域。

LoRA 的核心承诺是：

- **还原**：从来源图、来源配方、样例与触发词出发，尽可能重建可解释、可继续调整的生成状态。
- **定制**：在保持角色、风格或主体特征的同时，调整提示词、负面提示词、底模、挂载组合、权重、参考素材与生成参数。

训练和导入用于扩展可用 LoRA 资源，但不是唯一主线；用户即使不训练自己的 LoRA，也应能完成发现、理解、挂载、组合、还原和生成。

## 2. LoRA 域负责什么

- 发现与检查可用的公共 LoRA 资源，包括当前接入的 Civitai 与 Hugging Face 来源。
- 管理用户可用的 LoRA 资源，包括收藏、导入和训练产物。
- 将一个或多个 LoRA 挂载到本次生成，管理顺序、启用状态、权重与剩余容量。
- 明确 LoRA 与底模的兼容关系，帮助用户选择可执行的组合。
- 提供 LoRA 专属生成上下文：提示词、负面提示词、触发词、标签、来源配方、样例参考与真实可用参数。
- 支持“还原来源效果”和“保留特征后继续定制”两类核心工作流。
- 训练自定义 LoRA，并让训练产物回到“我的资源”和生成流程继续使用。
- 保存 LoRA 生成结果的来源面、挂载谱系和必要参数，使结果可追溯并进入长期资产体系。

## 3. LoRA 域不负责什么

| 相邻域       | 边界                                                                                                |
| ------------ | --------------------------------------------------------------------------------------------------- |
| Studio Image | 负责不以 LoRA 为中心的通用图片生成。Image 页面不为了“功能齐全”复制 LoRA 挂载、组合和训练工作台。    |
| Assets       | 负责生成结果与素材的长期归档、整理和复用。LoRA 可以产出资产，但不复制完整资产管理器。               |
| Gallery      | 负责公开作品的发现与浏览。LoRA 的公共资源库是模型资源发现，不等同于作品画廊。                       |
| Prompts      | 负责持久化的个人配方、模板与版本复用。LoRA 负责本次生成中的装配状态，不在域内另建一套通用提示词库。 |
| Canvas       | 负责节点编排与跨能力工作流。LoRA 可以成为画布能力或节点的输入，但不在工作台内复制画布编排。         |
| 执行基础设施 | Runner、provider、队列和回调负责真实执行。LoRA 页面只暴露已接通的能力，不用 UI 伪装尚未支持的参数。 |

## 4. 当前信息架构事实

- 主路由：`/studio/lora`。
- 顶层工作模式：**生成（Generate）/ 资源库（Library）/ 训练（Train）**。
- 资源库包含：**公共资源（Public / Community）** 与 **我的资源（Mine）**。
- 公共资源当前保留 Civitai 与 Hugging Face 的来源差异；不能为了视觉统一虚构相同字段或抹平真实能力差异。
- 当前默认 section 与 query 契约以 `src/constants/lora.ts` 和路由代码为事实源。

这组结构是当前业务事实，不是未来页面必须使用 tab、卡片网格或现有壳层的视觉命令。未来可以改变空间结构与交互组织，但不得丢失用户任务、可达性和 URL 状态契约。

## 5. 核心工作流

1. **发现并生成**：公共资源 → 查看来源、样例与触发信息 → 收藏或挂载 → 选择兼容底模 → 生成。
2. **从我的资源生成**：我的资源 → 挂载或组成 LoRA 栈 → 调整权重与触发词 → 生成。
3. **还原**：来源图或来源配方 → 恢复可用的 LoRA、底模、提示词与参数 → 对照样例生成。
4. **定制**：在还原状态上改变可变层，包括描述、负面约束、参考素材、挂载组合、权重与真实可用参数。
5. **训练后复用**：准备训练输入 → 训练 → 得到 LoRA 资源 → 进入我的资源 → 挂载并生成。

未来 UI 设计必须先证明这些工作流如何更清楚、更有辨识度，再讨论造型。不能只为当前组件换皮。

## 6. 领域词汇

| 术语          | 在 PixelVault 中的含义                                                                                                    |
| ------------- | ------------------------------------------------------------------------------------------------------------------------- |
| LoRA 资源     | 可被检查、收藏、导入、训练、管理或挂载的模型能力单元；不等同于一张作品图片。                                              |
| 收藏          | 把外部来源记录纳入“我的资源”以便再次发现和使用；不等同于 Prompts 的配方收藏。                                             |
| 挂载          | 将 LoRA 资源加入当前生成状态，并带有顺序、启用状态和权重。                                                                |
| LoRA 栈       | 当前生成中按顺序装配的一组 LoRA 挂载；需要表达容量与兼容关系。                                                            |
| 底模          | 实际执行所需的基础 checkpoint；可选范围受 LoRA family、架构和 runner 能力约束。                                           |
| 触发词        | 与 LoRA 资源相关、用于激活或引导特征的词组；它可进入提示词，但不等同于整份提示词。                                        |
| 来源配方      | 从外部作品或样例得到的提示词、负面提示词、模型与参数证据，用于理解和还原；只有用户明确保存后才成为 Prompts 域的持久配方。 |
| 样例参考      | 帮助用户理解 LoRA 效果或发起还原的媒体证据；不自动等同于生成时已提交的 reference input。                                  |
| LoRA 生成结果 | 由共享图片生成链路产出的 `Generation`，带有 LoRA 来源面与挂载谱系；长期管理归 Assets。                                    |
| 训练产物      | 用户训练完成后可进入“我的资源”并参与挂载与生成的 LoRA 资源。                                                              |

## 7. 不能破坏的业务约束

- LoRA 保持独立业务入口，不退化成 Image Studio 的一个高级参数区。
- 页面只能呈现后端、provider 与 runner 已真实接通的能力；不支持的参数必须隐藏、禁用或明确说明。
- 兼容关系、底模约束和执行失败必须可理解，不能只依赖颜色表达。
- Civitai、Hugging Face 和用户训练资源可以复用行为与组件 API，但来源字段和能力差异必须保真。
- LoRA 生成复用共享生成引擎与结果体系，不另造一套图片生成执行链。
- 生成结果必须保留足够的 LoRA lineage，且能进入 Assets；页面重构不能切断持久化和回放路径。
- 共享组件统一行为、状态、无障碍与响应式契约，不强制 LoRA 沿用其他域的皮肤。
- 所有用户可见新增文案须保持 en / ja / zh 三语就绪。

## 9. 拍板状态（2026-07-19 更新）

- ✅ **设计范围已确认**：`/studio/lora` 作为一个完整业务域建立一套域级视觉身份，覆盖 Generate、Library、Train。
- ✅ **一致性的边界已确认**：三个工作空间共享域级身份，但允许根据各自任务采用明显不同的空间结构，不要求复制同一页面骨架。
- ✅ **业务主次已确认**：Generate 是 LoRA 的执行核心；Library 负责发现、理解和管理可用于生成的资源，Train 负责生产新的 LoRA 资源，两者都应能自然回流 Generate。
- ✅ **日常入口已确认**：Library 是用户最常打开的空间，也是 `/studio/lora` 的默认入口；这与当前无 section 时进入 `COMMUNITY` 的代码行为一致。Library 是高频入口，Generate 是执行核心，两者不是同一个概念。
- ✅ **跨空间主流程已确认**：用户在 Library 确认使用某个 LoRA 后，应携带该 LoRA 的挂载状态直接进入 Generate，而不是要求用户再次查找或挂载。
- ✅ **Library 选择语义已确认**：普通卡片点击只打开详情；详情中的明确主动作“使用此 LoRA”不再弹二次确认，点击后立即挂载该 LoRA 并跳转 Generate。
- ⏸ 已有 LoRA 栈时的追加/替换、容量满和不兼容处理属于现行业务收口，不再作为视觉方向讨论的前置门槛；需要改功能时回到 `lora-workbench.md` 单独确认。
- ✅ **当前设计阶段已确认**：现有 LoRA 功能骨架足以进入 UI 设计，下一步直接确认视觉层级、域级性格与空间结构，不继续穷举功能规则。
- ✅ **视觉层级已确认**：Library 采用“效果优先、数据第二层”。样例图、风格差异与能力证据先获得注意力；底模、触发词、兼容性、热度等保持精确清晰，但不与效果内容争夺第一视线。进入 Generate 后，挂载组合与真实生成参数上升为主角。
- ✅ **参照类型已确认**：LoRA 借鉴 Claude.ai / ChatGPT **产品工作台**的设计方法——任务聚焦、排版主导、容器克制、控件按需出现；不借官网营销页的大标题叙事，也不把 LoRA 结构改造成聊天对话界面。
- ✅ **审美边界已确认**：拒绝模型市场式的花哨瀑布流、强装饰材质和过量视觉标签；LoRA 的亮眼来自比例、留白、信息节奏与少量精确细节，不来自堆叠卡片、pill 或颜色。
- ✅ **明暗基调已确认**：采用**克制的中性深色工作台**。以柔和炭灰和中性层级组织空间，避免纯黑压迫、大面积蓝黑浮层与高饱和强调色；辨识度主要来自排版、间距、比例、细分隔线和按上下文出现的控件，而不是装饰材质或颜色堆叠。
- ✅ **旧候选隔离已确认**：该基调不继承当前页面的全黑卡片/pill 造型，也不采用 Claude 草稿中的骨白纸面、朱砂和标本索引语言；两者都只能作为对照证据。
- ✅ **LoRA 顶部区域已确认**：未来 LoRA 设计删除内容区上方仅显示“工作台”的冗余顶栏及重复账号入口，LoRA 内容直接从 Generate / Library / Train 工作模式开始。全站左侧导航与账号入口仍由应用外壳负责，本决定不授权重构其他业务域的全局 shell。
- ✅ **三方向比较载体已确认**：先以 Library 作为三个结构方向的共同比较页面。三个方向必须使用同一组 LoRA、样例图、搜索/筛选条件和选中详情状态，只比较空间结构、注意力路径与信息揭示方式；不得用不同内容制造方向差异。
- ✅ **比较范围已确认**：三方向阶段不同时展开 Generate、Library、Train 三套完整方案。先在高频默认入口 Library 验证“效果优先、数据第二层”，选定方向后再制作正式关键切片，并把域级身份延展到 Generate 与 Train。
- ✅ **正式候选组已确认**：Library 进入比较的三个结构方向为“聚焦浏览”“搜索工作台”“对比工作台”。三者分别验证单列大幅效果浏览、搜索+结果+上下文检查，以及多候选并列比较；当前只是确认比较对象，尚未选定最终方向。
- ✅ **Library 筛选层级已确认**：类型与底模是高频选择条件，三个方向都必须在默认页面中直接可见或一眼可达，不能把两者一起收进无状态提示的通用“筛选”按钮。公开/我的、来源、排序与安全仍需保留，但可以使用更低的视觉层级组织。
- ✅ **Library 分页已确认**：三个方向必须表现真实分页，包括上一页、当前页与下一页；分页不能因为概念稿只展示首屏而消失。
- ✅ **Library 结构选向已确认**：owner 选择“聚焦浏览”作为后续关键切片的结构基础，即单列宽幅效果流、选中项原位置展开详情；“搜索工作台”保留为对照，不再与选中方向并行深化，“对比工作台”不进入本轮关键切片。
- ✅ **Library 筛选控件形态已确认**：类型与底模保持一等可见性，使用两个独立下拉/组合框并持续显示当前值；底模选择需要支持检索大量候选。公开/我的、来源、排序与安全仍保留为较低层级控制。
